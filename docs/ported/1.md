# Memory Port Comparison: Current Project vs TencentDB

## Scope

This document compares the **core memory system** in the current project against `TencentDB/` and focuses on what was fully ported, partially ported, or not ported.

### Explicit exclusions

This comparison **excludes**: 

- Telegram-specific implementation details
- TencentDB host/plugin wiring that is only relevant to Telegram/OpenClaw integration
- the memory adapter layer borrowed/used from TencentDB rather than the core memory behavior itself

The goal is to answer this question precisely:

> Which TencentDB memory capabilities are truly present in the current project, and which ones are still reduced, weakened, or missing?

---

## Executive summary

The current project **did port the major memory concepts** from TencentDB:

- layered memory (`L0` / `L1` / `L2` / `L3`)
- retrieval before generation
- memory extraction pipeline
- persona generation
- offload / evidence capture
- task memory / task canvas
- hybrid retrieval primitives
- lineage / provenance links

However, the port is **not behaviorally equivalent** to TencentDB.

The current project is best described as:

> **an agent-integrated memory service with scheduled consolidation**

TencentDB is better described as:

> **a fuller stateful memory runtime with richer dedupe, scene management, prompt injection strategy, and per-session pipeline behavior**

### Strongest parts of the port

- layered architecture exists
- recall exists and is used in live generation
- L1/L2/L3 pipeline exists
- offload exists
- hybrid search exists
- lineage/provenance exists

### Weakest or most incomplete parts of the port need to port

- semantic L1 dedupe
- scene-based L2 management
- incremental scene-aware L3 persona updates
- prompt-cache-aware recall injection
- per-session pipeline scheduling and flush behavior
- full offload runtime engine
- some config propagation into the live bootstrap

## 17. Final conclusion

The current project is **not missing the whole memory port**.

It has a real and substantial TencentDB-derived memory architecture, including:

- layered memory
- recall
- pipeline processing
- persona generation
- offload
- task evidence
- provenance

But it is **not a full behavioral port**.

The biggest differences are not in whether features exist at all, but in **how deeply and robustly they behave**.

### Best one-sentence judgment

> The current project ported TencentDB’s memory architecture at the feature-concept level, but not TencentDB’s deeper runtime semantics.

### Best practical summary

If you want to make this project feel much closer to TencentDB, the highest-value work would be:

1. bring over TencentDB-grade semantic L1 dedupe
2. bring over TencentDB-grade scene-based L2 management
3. bring over TencentDB-grade session-aware pipeline behavior
4. bring over TencentDB-grade incremental L3 persona updates
5. split recall into stable vs dynamic prompt context
6. tighten offload runtime behavior
7. fix the `src/index.ts:63` config propagation gap
